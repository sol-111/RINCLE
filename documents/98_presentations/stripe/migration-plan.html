Migration Plan

Stripe 移行方針

4/21・5/1 定例MTGでの合意に基づく、Pay.jpからStripeへの移行計画。一括回収モデル・予約時決済・店頭決済廃止を実現します。

決済モデル

PEDALSTANDARD 一括回収 → 手数料差引後に店舗へ振込

決済タイミング

予約時に即時決済（来店時の決済を廃止）

利用料

プラットフォーム利用料 15%

移行フェーズ

Phase 0〜3 の4段階で段階的に移行

1 なぜ移行するのか

現行の Pay.jp（プラットフォーム型）からStripe（シンプル決済）へ移行する背景です。

現行（Pay.jp）の課題

店舗ごとにアカウントが必要 Pay.jpの審査を通す必要があり、店舗登録に手間と時間がかかる
手数料の徴収が煩雑 Pay.jp上で10%自動差引 + 残り5%を紙の請求書で別途回収。未払いリスクあり
セキュリティ上の問題 決済関連のAPIキーや顧客情報がデータベース上で閲覧可能な状態
CVC検証がされない カード登録時にセキュリティコードが誤っていても登録できてしまい、決済時にエラーになる

Stripe移行で解決すること

店舗アカウント不要 PEDALSTANDARDが一括回収するため、店舗に決済アカウントを作らせる必要がない
手数料回収がシンプル 全額を先に回収し、手数料差引後に店舗へ振込。未払いリスクがゼロ
3Dセキュア標準対応 追加開発なしで本人認証が自動的に走る。不正利用を防止
CVC検証あり カード登録時にセキュリティコードを検証。誤りがあれば登録できない

2 新しいお金の流れ

PEDALSTANDARDが売上を一括回収し、手数料を差し引いて店舗へ振り込む仕組みです。

エンドユーザー予約時にカード決済
→
Stripeカード手数料 3.6% 差引
→
PEDALSTANDARD一括回収
→
店舗利用料15%差引後に振込

1万円の取引の場合

エンドユーザーの支払い10,000円
Stripe 手数料（3.6%）-360円
PEDALSTANDARD 受取額9,640円
プラットフォーム利用料（15%）-1,500円
店舗への振込額8,140円

現行との比較

項目 現行（Pay.jp） 新（Stripe）
売上の受取 店舗がPay.jpから直接受取 PEDALSTANDARDが一括回収
利用料の徴収 10%自動差引 + 5%を紙請求 15%を差引後に振込（紙請求なし）
未払いリスク あり（紙請求5%分） なし
店舗のアカウント作成 必要（審査あり） 不要
店頭決済（現金） あり 廃止（カード決済のみ）

3 決済フロー

各シーンでの決済の流れです。

予約時決済（通常バイク）

エンドユーザーが予約

バイクを選択し、カード情報を入力して予約を確定

即時決済

予約と同時にカードから全額引き落とし。3Dセキュア認証が自動で走る

決済エラーの場合 → エラー表示。別のカードで再試行

予約確定・在庫確保

決済完了と同時に予約が確定。バイクの在庫が確保される

ポイント：現行は来店時に店舗が「ライド開始」ボタンを押して決済していますが、新フローでは予約時に決済が完了します。来店時に店舗がボタンを押す必要はなく、時間軸で自動的に進行させることも検討可能です。（5/1 MTG）

キャンセル時

キャンセル操作

エンドユーザーまたは店舗がキャンセル

返金処理

キャンセルポリシーに基づき、全額または一部を返金

予約キャンセル・在庫開放

バイクの在庫が開放され、他のお客様が予約可能に

要決定：予約時に全額決済するため、キャンセル時は返金が発生します。「何日前まで全額返金」「当日キャンセルは○%」などのキャンセルポリシーの策定が必要です。また、キャンセル時の返金手数料をユーザーに負担させるかどうかも検討が必要です。（5/1 MTG）

延長時

店舗が延長操作追加料金を確定
→
追加決済登録済みカードを再利用
→
延長情報を更新

現場での追加オプション（ヘルメット、車両保険等）

店舗がオプション選択

ヘルメット、車両保険など。現場でお客様と相談の上、追加を決定

追加決済

登録済みのカードで追加料金を即時決済

決済失敗の場合 → ユーザーに別のカードを登録してもらい再試行

オプション情報を更新

決済完了後、予約情報にオプション内容と金額を記録

現状：オプション追加は現在リンクル外（現場で別途現金決済等）で処理されており、リンクルの売上データに反映されません。システム化することで売上の正確な把握が可能になります。車両保険については、高級バイクを中心に現場での加入案内が発生する見込みです。（5/1 MTG）

高級車のデポジット（将来対応）

予約時：通常料金を即時決済

通常のレンタル料金はその場で全額引き落とし

貸出日：デポジット分の枠を仮押さえ

例：10万円分の利用枠を確保。お金は引き落とさない（最大7日間有効）

返却時

問題なし → 枠を解放（何も起きない）
車体損傷等 → デポジット分を引き落として回収

4 店舗への振込フロー

PEDALSTANDARDから各店舗への振込の流れです。

月末に売上集計各店舗の決済済み売上
→
手数料差引Stripe手数料 + 利用料15%
→
店舗へ振込
項目 現行 新
振込元 Pay.jpから各店舗へ直接 PEDALSTANDARDから各店舗へ
利用料の回収 10%自動差引 + 5%を紙請求書で別途 振込前に15%を差引（請求書不要）
回収漏れ 紙請求5%分に未払いリスクあり リスクなし（先に全額回収済み）
要決定：振込頻度（月1回 or 月2回）、振込手数料の負担者（PEDALSTANDARD or 店舗）を決める必要があります。

5 領収書の発行

一括回収モデルでは、PEDALSTANDARDがユーザーに対して領収書を発行する必要が出てくる可能性があります。（5/1 MTG）

項目 現行 新
お金のやり取りの当事者 ユーザー ↔ 店舗 ユーザー ↔ PEDALSTANDARD
領収書の発行元 店舗 PEDALSTANDARD（要確認）
発行手段 店舗が紙で発行 Stripeのレシート自動送信機能が利用可能
Stripeの機能：Stripeには決済完了時にレシートメールを自動送信する機能が標準搭載されています。法的な要件を満たすかは別途確認が必要ですが、基本的なレシート発行はシステム側で自動化可能です。

7 移行スケジュール

4段階に分けて段階的に移行します。

Phase 0 — 即時対応（〜GW前）

現行システムの緊急修正

  • プラットフォーム手数料を 10% → 15% に変更
  • 不具合の洗い出し・修正（ほぼ完了）
  • 新データベース設計・モジュール設計（今週完了予定）
  • テスト環境の本番キー使用を修正
  • データベースのセキュリティ対応
  • 延長決済の処理不備修正
  • カード選択の不具合修正
Phase 1 — 方針確定（5/8 MTG〜）

決済サービス決定・設計

  • Pay.jp vs Stripe 比較資料で方針決定（5/8 MTG）
  • 弁護士回答の確認（会計処理・責任所在）
  • Stripeアカウント開設
  • 決済フローの詳細設計
  • キャンセルポリシーの確定
  • 領収書発行方法の確定
Phase 2 — 実装（5/11〜）

検索改修 + Stripe決済の実装

  • 検索周りの大幅改修（最優先・段階リリース）
  • 予約時決済フローの実装
  • キャンセル・返金フローの実装
  • 延長・追加オプション決済の実装
  • 高級車デポジットの実装（後回し可）
  • 管理画面への決済ステータス表示
Phase 3 — 移行・切替

本番切替・運用開始

  • テスト環境での検証
  • 既存ユーザーへのカード再登録案内
  • 本番環境への切替
  • Pay.jp 店舗アカウントの廃止
  • 店舗への振込フロー運用開始

8 決め残し・要確認事項

方針確定に向けて、決定が必要な事項です。

PEDALSTANDARD 様 側で決定が必要

  • キャンセルポリシーの策定
    何日前まで全額返金、当日キャンセルは何%等
  • 弁護士回答の共有
    一括回収時の会計上の扱い・責任所在
  • 店舗への振込頻度
    月1回 or 月2回
  • 振込手数料の負担者
    PEDALSTANDARD or 店舗
  • 追加オプション（ヘルメット等）のシステム化可否
  • 車両保険の料金体系・加入タイミング
    現場での加入案内が発生する見込み（5/1 MTG）
  • キャンセル時の返金手数料をユーザーに負担させるか
    5/1 MTG時点で未確定
  • 領収書の発行義務・発行元の確認
    一括回収モデルではPEDALSTANDARDが発行元になる可能性（5/1 MTG）

開発チーム（弊社）側で確認

  • Stripe手数料率の最終確認
    契約プランにより変動の可能性
  • Bubble連携プラグインの選定・検証
  • 既存ユーザーのカード再登録方法
    カード情報はセキュリティ上、直接移行不可
  • 決済通知（Webhook）の設計
  • 3Dセキュア認証フローの設計
  • 来店時のオペレーション設計
    予約時決済で来店時の操作が不要に。自動進行 or 確認ボタン（5/1 MTG）
  • 決済失敗時の別カード登録フロー
    オプション追加時等の失敗時、ユーザーに別カード登録を促すUI（5/1 MTG）
