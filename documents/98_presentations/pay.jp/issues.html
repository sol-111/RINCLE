1
開発環境で本番キーが使用されている
1 1st
🔍 現象
APIコネクタの設定が本番キーになっており、開発環境でも本番向けのAPIが呼ばれる
💥 影響
テストができない。開発中の操作が本番データに影響する可能性
✅ 対応
開発用ワークフロー / 本番用ワークフローを分けるか、環境変数で切り替える仕組みを導入
💡
優先理由
全ての開発・テスト作業の前提。これがないと他の修正の検証もできない
2
DBが完全オープン（セキュリティ）
2 2nd
🔍 現象
DBは誰でも閲覧可能な状態
💥 影響
顧客情報・決済情報が外部から参照できるリスク
✅ 対応
プライバシールールを設定し、適切なアクセス制御を行う
💡
優先理由
事故が起きたら取り返しがつかない。放置期間が長いほどリスク増大
3
プラットフォーム手数料が契約と異なる（10% → 15%）
3 3rd
🔍 現象
契約上のRINCLE取り分は売上の15%だが、現在のシステムでは10%に設定されている。全ての決済で5%分の収益が失われている
💥 影響
1万円の決済あたり500円の損失。取引が発生するたびに損失が拡大し続けている
✅ 対応
Pay.JPのプラットフォーム手数料の設定値を10% → 15%に変更する（設定1箇所の変更のみ）
💡
優先理由
契約通りの収益が得られていない。修正は設定変更のみで工数最小、即日対応可能
🔧 開発者向け詳細
設定値の場所
場所現在の値修正後
Bubble Option Set platform_fee 10 15

Bubble管理画面 → Data → Option Sets → platform_fee → 「デフォルト」の number 値を 10 → 15 に変更するだけ。

platform_fee の計算式
platform_fee = デフォルト's number / 100 × amount 例: 予約金額 10,000円の場合 現状: 10 / 100 × 10,000 = 1,000円（10%） 修正後: 15 / 100 × 10,000 = 1,500円（15%）
この計算式が使われている全ワークフロー
ワークフロートリガー画面
Charge_by_car_id 「ライド開始」ボタン shop_admin
Charge_by_car_id 「ライド終了」ボタン（延長料金） shop_admin
Charge_by_car_id ステータス変更（管理者決済） admin

3箇所すべてで同じ Option Set を参照しているため、Option Set の値を1箇所変えれば全ワークフローに反映される。

Pay.JP に送信されるパラメータ（Charge_by_car_id）
customer: 予約情報の cus_id amount: 合計金額（or 延長料金） currency: "jpy" tenant: 店舗の tenant_id platform_fee: デフォルト's number / 100 × amount ← ここが10%
お金の流れの変化（1万円の場合）
受取人現状（10%）修正後（15%）
RINCLE（PEDALSTANDARD）1,000円1,500円
Pay.JP（カード手数料 約3.5%）約350円約350円
店舗（ネット入金）約8,650円約8,150円
注意事項
  • Option Set の値変更は即座に全ワークフローに反映される（デプロイ不要）
  • 変更前の決済（既に charge 済み）には影響しない。変更後の新規決済から適用
  • 店舗のネット入金が減るため、店舗への事前通知の要否をクライアントに確認
4
延長決済の charge_id が保存されていない
4 4th
🔍 現象
shop_admin の延長ワークフロー（bTdud / ライド終了ボタン）で Charge_by_car_id を実行した後、返却された charge_id を予約データに保存していない
💥 影響
延長分の返金時に __charge_id が空のため、返金できない可能性がある
✅ 対応
延長決済成功後に __charge_id フィールドへ charge_id を保存する処理を追加
💡
優先理由
延長が発生した時点で返金不能になる（現時点で延長実績ゼロのため実害は未発生）
🔧 開発者向け詳細
DBフィールド構成（予約テーブル）
フィールド用途現状
charge_id通常決済（ライド開始時）の決済ID保存されている ✅
__charge_id延長決済（ライド終了時）の決済ID空のまま ❌
__charge_id1返金用charge_id—
現状のワークフロー（ライド終了ボタン / WF: bTdud）
① 延長料金があるか確認 ↓ ② Pay.JP に追加決済を依頼（Charge_by_car_id API） ↓ ③ Pay.JP が決済成功 → charge_id を返す ↓ ④ ← ここで __charge_id に保存すべきだが、していない ↓ ⑤ ステータスを「決済済み」に更新
返金時に何が起きるか
管理者が「返金あり」をクリック ↓ 通常分の返金 → charge_id で Pay.JP に返金依頼 → 成功 ✅ ↓ 延長分の返金 → __charge_id で Pay.JP に返金依頼 → 空なので返金できない ❌
修正内容

対象: shop_admin「ライド終了」ボタンのワークフロー（bTdud）

現状: Charge_by_car_id → make change 予約（ステータス: 決済済み） 修正後: Charge_by_car_id → make change 予約（__charge_id = API response の charge_id） → make change 予約（ステータス: 決済済み）
補足
  • 延長の実績は現時点でゼロ。実害は出ていないが、1件でも発生した時点で返金不能になる
  • 延長料金はライド開始時の決済とは別の charge として処理される
  • charge_id（通常）と __charge_id（延長）を個別に返金する設計
5
デフォルトカードの設定がドロップダウン選択と不一致
5 5th
🔍 現象
予約確定時に設定されるデフォルトカードは「取得した顧客カードの1つ目」であり、ユーザーがドロップダウンで選択したカードではない
💥 影響
ユーザーが選んだカードとは異なるカードで決済される可能性がある
✅ 対応
ドロップダウンで選択されたカードをデフォルトカードに設定するよう修正
💡
優先理由
クレーム直結。複数カード登録ユーザーで発生する
🔧 開発者向け詳細
現状のワークフロー（index / 予約確定ボタン）
① ユーザーがDropdownでカードを選択 ↓ ② 「予約を確定する」ボタンをクリック ↓ ③ API get_client_action で顧客情報を取得 ↓ ④ API account_register_default_card で1つ目のカードをデフォルトに設定 ← Dropdownの選択値を無視して、取得したカード配列の先頭を使っている ↓ ⑤ 予約レコード作成（ステータス: 来客待ち）
問題の原因

account_register_default_card の default_card パラメータに、Dropdownの値ではなく get_client_action のレスポンスから取得したカード配列の最初の要素が渡されている。

修正内容
項目現状修正後
default_card パラメータ get_client_action の cards[0].id Dropdown の value（ユーザー選択カードID）
修正箇所
  • ページ: index（ユーザー予約画面）
  • ワークフロー: 「予約を確定する」ボタンの click イベント内
  • 対象API: account_register_default_card
  • 変更: default_card に Dropdown の value を渡す
再現条件

ユーザーが2枚以上のカードを登録しており、かつ 2枚目以降のカードを選択して予約した場合に発生。カード1枚のユーザーでは問題は起きない。

6
オーソリ（与信確認）が未実装
6 6th
🔍 現象
クライアント提供の「RINCLEクレジット支払いフロー.pdf」ではオーソリ→キャプチャモデルが定義されているが、実際には実装されていない。予約時にオーソリなし、ライド開始時に直接課金している
💥 影響
クライアントは実装済みと認識している可能性が高い。未回収リスクの抑制が機能していない
✅ 対応
オーソリ→キャプチャモデルを実装（capture=false / expiry_days=60）。最大予約先も60日に制限、営業カレンダーも60日分に縮小
💡
優先理由
クライアントが想定している仕様と実装が乖離している（仕様未達）
🔧 開発者向け詳細
現状（As-Is）の決済フロー
予約確定時（index）: カード設定のみ。課金もオーソリもなし ライド開始時（shop_admin / ライド開始ボタン）: POST /v1/charges（capture=true ← デフォルト） → 全額を直接課金（オーソリなし）
あるべき姿（To-Be）の決済フロー
予約確定時（index）: POST /v1/charges に capture=false を追加 → オーソリ（与信枠確保）のみ。お金は動かない → 返却された charge_id を予約データに保存 ライド開始時（shop_admin / ライド開始ボタン）: POST /v1/charges/:id/capture → 予約時のオーソリを確定（キャプチャ） → ここで初めてお金が動く キャンセル時（admin）: オーソリ解放のみ（返金処理不要）
API パラメータの変更
タイミング現状のAPI修正後のAPI
予約確定時 なし（カード設定のみ） POST /charges capture=false, expiry_days=60
ライド開始時 POST /charges（全額課金） POST /charges/:id/capture
キャンセル時 POST /charges/:id/refund オーソリ自動失効（API不要）
追加で必要な修正
  • 営業カレンダー: 365日分 → 60日分に縮小（オーソリ有効期限と整合）
  • 最大予約先: 60日以内に制限
  • 二重キャプチャ防止: キャプチャ済みフラグのチェックを追加
  • キャプチャ失敗時: 別カード再決済 or 現金フォールバック（要設計）
影響範囲
ページワークフロー変更内容
index予約確定ボタンオーソリAPI追加 + charge_id保存
shop_adminライド開始ボタンCharge → Capture に変更
adminキャンセル処理Refund → 不要に（オーソリ解放）
adminステータス変更Charge → Capture に変更
🚨
即時対応の6件は着手順に沿って進めてください。まず #1（本番キー分離）→ #2（DB公開）で安全な環境を整え、#3（手数料15%修正）で収益損失を止めた上で、#4（charge_id保存）→ #5（カード選択）→ #6（オーソリ実装）の順に着手します。

エラーコード意味ユーザー向けメッセージ

10
PEDALSTANDARD の 15% 利用料請求の仕組みが未定
低優先
🔍 現象
PEDALSTANDARDから販売店への利用料15%の請求方法が決まっていない
📖 現在の platform_fee
10%（Pay.JP上のプラットフォーム手数料）
📖 検討案
platform_fee を 0 に寄せて、15% は Pay.JP とは別に毎月末請求する運用
✅ 対応
ビジネス要件を確定させた上で実装方針を決定
🔧 開発者向け詳細
選択肢の比較
方式Pay.JP の platform_feeRINCLE取り分リスク
A. platform_fee を15%にする 15%（自動差引） Pay.JPが自動で分配 低（自動徴収）
B. platform_fee を0にして月末請求 0%（差引なし） 月末に請求書で15%回収 高（未払いリスク）
方式Bを選ぶ場合に必要な実装
  • Option Set platform_fee を 15 → 0 に変更
  • 月末売上集計ワークフローの新規実装
  • 請求データテーブル（Monthly_Invoice）の新規作成
  • admin画面に請求管理セクション追加
  • 締め日・入金期限・督促ルールの策定
現時点の方針

まず #3 で platform_fee を15%に修正（方式A）して収益損失を止める。月末請求化（方式B）は別フェーズで検討。
