# Pay.jp vs Stripe 比較資料

> 2026/5/1 MTG を受けて作成。5/8 定例MTGでの方針決定に使用。
> リンクルの新決済モデル（PEDALSTANDARD一括回収・予約時決済・店頭決済廃止）を前提に比較。

---

## 1. 前提：新しい決済モデル

4/21・5/1 MTGで合意した方針:

- **店頭決済（現金）を廃止**し、カード決済オンリーにする
- **PEDALSTANDARDが売上を一括回収**し、手数料を差し引いて店舗へ振込
- **予約時に即時決済**する（オーソリ→キャプチャ方式ではない）
- 店舗にPay.jp/Stripeのアカウントを作らせない
- 高級車のデポジットオーソリは将来対応（貸出日当日、7日以内）

```
エンドユーザー → 決済サービス → PEDALSTANDARDが一括受取
                                        ↓
                              手数料差引後、店舗へ振込
```

| | Pay.jp（Visa/Master） | Pay.jp（JCB等） | Stripe |
|--|---|---|---|
| 決済手数料 | 300円 | 360円 | 360円 |
| PEDALSTANDARD受取 | 9,700円 | 9,640円 | 9,640円 |

## 5. 「シンプルな決済のみ」で使う場合の比較

新モデルでは、Pay.jp も Stripe も **Platform型の機能は使わない**。
純粋に「カードで決済→PEDALSTANDARDの口座に入金」だけ。

### Pay.jp を継続する場合

**やること:**
- 既存のPlatform API呼び出しを全て撤去（テナント作成・審査・platform_fee等）
- `Charge_by_car_id` から `tenant` と `platform_fee` パラメータを削除
- 店舗登録フローからPay.jpアカウント作成の8ステップを削除
- 3Dセキュアを別途実装（Pay.jp v2 の 3DS API）
- CVC検証の問題は残る（Pay.jpの仕様上、登録時の検証不可）

**メリット:**
- 既存のAPI Connector定義（15本）のうち一部を流用可能
- 既存の顧客データ（`cus_xxxxx`）がそのまま使える → カード再登録不要の可能性

**デメリット:**
- 3Dセキュア未対応のまま or 別途実装コストが発生
- CVC未検証の問題が根本的に解決しない
- Bubble公式プラグインがない（APIコネクタで手動管理）
- Platform型→通常型への切り替えでPay.jp側の契約変更が必要な可能性あり

### Stripe に移行する場合

**やること:**
- Stripe アカウント開設（PEDALSTANDARD名義）
- Bubble公式 Stripe プラグイン導入
- PaymentIntent / PaymentMethod ベースで決済フロー実装
- 既存ユーザーにカード再登録を依頼

**メリット:**
- 3Dセキュアが標準対応（追加実装不要）
- カード登録時にCVC検証される（決済失敗リスク低減）
- Bubble公式プラグインで実装がシンプル
- レシートメールの自動送信機能あり
- 将来のApple Pay/Google Pay対応が容易
- ドキュメント・事例が豊富

**デメリット:**
- 既存ユーザーのカード情報が移行不可（再登録が必要）
- Pay.jpの既存API定義を全て置き換える必要がある
- JCB/Amex利用時の手数料差はなし（Visa/Masterだと若干高い）

## 6. 開発工数の比較

### 共通でやること（どちらを選んでも必要）

| タスク | 内容 |
|--------|------|
| 店舗登録フローの簡略化 | Pay.jpアカウント作成の8ステップ削除 |
| 予約フローの変更 | 予約時即時決済に変更 |
| キャンセル・返金フローの実装 | キャンセルポリシーに基づく返金処理 |
| 延長決済フローの修正 | 追加決済の実装 |
| 管理画面の更新 | 決済ステータス表示・返金操作 |
| DB設計変更 | 決済関連テーブルの見直し |

### Pay.jp 継続の場合の追加工数

| タスク | 工数感 |
|--------|--------|
| Platform API 撤去 | 小（削除するだけ） |
| 通常決済モードへの切り替え | 小〜中（契約変更 + API呼び出し修正） |
| 3Dセキュア実装 | **中〜大**（Pay.jp v2 API + フロントエンド対応） |
| 既存API定義の整理（15本→必要分に削減） | 小 |

### Stripe 移行の場合の追加工数

| タスク | 工数感 |
|--------|--------|
| Stripeアカウント開設・設定 | 小 |
| Bubbleプラグイン導入・設定 | 小 |
| 決済フロー実装（PaymentIntent） | 中（ただしプラグインでシンプル） |
| 既存Pay.jp API定義の撤去 | 小（削除するだけ） |
| ユーザーへのカード再登録案内 | 小（メール送信 + 再登録UI） |

### 工数の結論

**ほぼ同等**。共通タスクが大部分を占めるため、決済サービスの違いによる工数差は小さい。
ただし、Pay.jp で 3Dセキュアを別途実装する場合はその分が上乗せされる。

## 7. リスク比較

| リスク | Pay.jp | Stripe |
|--------|--------|--------|
| **CVC未検証による決済失敗** | **あり**（仕様上回避不可） | なし |
| **3Dセキュア未対応** | **あり**（別途実装が必要） | なし（標準対応） |
| **既存ユーザーへの影響** | なし（カード再登録不要） | **あり**（カード再登録が必要） |
| **契約変更リスク** | **あり**（Platform→通常への変更が必要か要確認） | なし（新規契約） |
| **Bubbleアップデートへの追従** | 手動管理（APIコネクタ） | プラグインで自動追従 |

## 8. 結論・推奨

### 推奨: Stripe

| 判断軸 | 結論 |
|--------|------|
| 開発工数 | ほぼ同等（3Dセキュア考慮でStripeが若干有利） |
| セキュリティ | **Stripe が優位**（3Dセキュア標準、CVC検証あり） |
| Bubble連携 | **Stripe が優位**（公式プラグイン） |
| 手数料 | Pay.jp がVisa/Masterで若干安い（差は小さい） |
| 将来性 | **Stripe が優位**（Apple Pay、領収書、豊富なAPI） |
| 移行リスク | **Pay.jp が有利**（カード再登録不要） |
| 開発チームの慣れ | **Stripe が優位**（弊社がStripeに慣れている） |

**唯一のデメリットであるカード再登録**は、現時点のユーザー数が少ないため影響は限定的。
ユーザー数が増えてからの移行はさらにコストが上がるため、早期に切り替える方が合理的。

### カード再登録の対応案

1. アプリ内で「決済方法が変わりました。カードを再登録してください」とバナー表示
2. 次回予約時にカード登録を必須にする（未登録なら予約に進めない）
3. 既存の予約で未決済のものがあれば、個別にメールで案内
