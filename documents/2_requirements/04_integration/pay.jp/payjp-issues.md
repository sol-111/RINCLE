# Pay.JP 連携 修正一覧

> Yanoさんの調査報告（2026-04-16）および `rincle.bubble` 解析結果に基づく。

## 対応方針

**目標: まず運用上問題ない状態にする。**

- **即時対応（5件）**: 運用が壊れる / 仕様未達 / 事故リスクがあるもの → 着手順: #3 → #2 → #1 → #5 → #4
- **改善時に対応（3件）**: 改善フェーズでまとめて対応
- **低優先（3件）**: 運用に影響なし。余裕があれば対応

---

## 即時対応

### 3. 開発環境で本番キーが使用されている 【着手順: 1番目】

- **現象:** APIコネクタの設定が本番キーになっており、開発環境でも本番向けのAPIが呼ばれる
- **影響:** テストができない。開発中の操作が本番データに影響する可能性
- **対応:** 開発用ワークフロー / 本番用ワークフローを分けるか、環境変数で切り替える仕組みを導入
- **優先理由:** 全ての開発・テスト作業の前提。これがないと他の修正の検証もできない

### 2. DBが完全オープン（セキュリティ）【着手順: 2番目】

- **現象:** 現状DBは誰でも閲覧可能な状態
- **影響:** 顧客情報・決済情報が外部から参照できるリスク
- **対応:** プライバシールールを設定し、適切なアクセス制御を行う
- **優先理由:** 事故が起きたら取り返しがつかない。放置期間が長いほどリスク増大

### 1. 延長決済の charge_id が保存されていない 【着手順: 3番目】

- **現象:** shop_admin の延長ワークフロー（`bTdud` / ライド終了ボタン）で Charge_by_car_id を実行した後、返却された charge_id を予約データに保存していない
- **影響:** 延長分の返金時に `__charge_id` が空のため、返金できない可能性がある
- **対応:** 延長決済成功後に `__charge_id` フィールドへ charge_id を保存する処理を追加
- **優先理由:** 延長が発生した時点で返金不能になる

### 5. デフォルトカードの設定がドロップダウン選択と不一致 【着手順: 4番目】

- **現象:** 予約確定時に設定されるデフォルトカードは「取得した顧客カードの1つ目」であり、ユーザーがドロップダウンで選択したカードではない
- **影響:** ユーザーが選んだカードとは異なるカードで決済される可能性がある
- **対応:** ドロップダウンで選択されたカードをデフォルトカードに設定するよう修正
- **優先理由:** クレーム直結。複数カード登録ユーザーで発生する

### 4. オーソリ（与信確認）が未実装 【着手順: 5番目】

- **現象:** クライアント提供の「RINCLEクレジット支払いフロー.pdf」ではオーソリ→キャプチャモデルが定義されているが、実際には実装されていない。予約時にオーソリなし、ライド開始時に直接課金している
- **影響:** クライアントは実装済みと認識している可能性が高い。未回収リスクの抑制が機能していない
- **対応:** オーソリ→キャプチャモデルを実装（`capture=false` / `expiry_days=60`）。最大予約先も60日に制限、営業カレンダーも60日分に縮小
- **優先理由:** クライアントが想定している仕様と実装が乖離している（仕様未達）

## 改善時に対応

### 6. エラーハンドリングの改善

- **現象:** APIエラー時は JavaScript の `alert()` で生のエラーメッセージが表示される
- **影響:** ユーザーが困惑し、対処方法がわからない
- **対応:**
  - エラーパターンを書き出す
  - APIコネクタでエラー時もワークフローが止まらないよう設定
  - エラーコードに応じたユーザー向けメッセージを表示

### 7. エラー通知・ログの仕組みがない

- **現象:** APIエラーが発生しても管理者・開発者に通知がこない。サーバーログは最短2週間で消える
- **影響:** エラーの検知・調査ができない
- **対応:**
  - エラー発生時にメール送信する仕組みを追加
  - 永続的なログ保存の仕組みを導入

### 8. 店舗招待URLのセキュリティ

- **現象:** shop_form への招待URL を知っていれば誰でも店舗登録できる
- **影響:** 不正な店舗登録のリスク
- **対応:** 招待メール送信時にパラメータとしてメアドか招待アカウントをDB保存し、unique_id で認証する

## 低優先

### 9. PEDALSTANDARD の 15% 利用料請求の仕組みが未定

- **現象:** PEDALSTANDARDから販売店への利用料15%の請求方法が決まっていない
- **現在の platform_fee:** 10%（Pay.JP上のプラットフォーム手数料）
- **検討案:** platform_fee を 0 に寄せて、15% は Pay.JP とは別に毎月末請求する運用
- **対応:** ビジネス要件を確定させた上で実装方針を決定

### 10. テナント登録時の銀行口座情報

- **現象:** テナント登録時（shop_form）に銀行口座情報を送信・DB保存しているが、Marketplace型では必須ではない
- **影響:** 不要なデータがDBに保存されている
- **対応:** Bubble側で使う必要がないため、DB保存をやめる

### 11. Webhook名のtypo

- **現象:** `payjp_webhook_update_talent` は `tenant` の typo
- **影響:** 動作には影響なし（Bubble内部IDで紐づいているため）
- **対応:** 可能であれば名称を修正（優先度低）
