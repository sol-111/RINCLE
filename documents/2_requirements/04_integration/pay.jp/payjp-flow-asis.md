# Pay.JP 決済連携仕様書（現状フロー）

## サマリ

- Rincleの決済は **Pay.JP** という外部の決済サービスを使っている
- お金の流れ: **お客さん → Pay.JP → 店舗**（Rincleは間に入ってプラットフォーム手数料10%をもらう。カード手数料 約3.5% は店舗負担）
- 店舗がカード決済を受けるには、Pay.JPの **審査に合格する必要がある**（Rincle側に承認機能はない）
- お客さんはカードを1回登録すれば、以降の予約でそのまま使える

## 概要

Rincleは「お客さんが店舗に予約してカードで払う」ための仕組みを、**Pay.JP** という決済サービスを使って実現しています。

登場人物は3者です:

| 登場人物 | 役割 | 例えるなら... |
|---|---|---|
| **Rincle** | 全体を取りまとめる（プラットフォーム） | ショッピングモールの運営会社 |
| **店舗** | サービスを提供してお金を受け取る | モール内のお店 |
| **お客さん** | 予約してお金を支払う | お店のお客さん |

Pay.JPは「ショッピングモールのレジシステム」のような存在で、カード決済の処理やお金の分配を代行してくれます。

## 決済フロー

> このセクションでは、Pay.JP決済に関わる全フローを「誰が・何をするか」の視点で説明します。

### 1. 店舗の登録〜審査フロー

店舗がRincleで決済を受け取れるようになるまでの流れです。

```
【誰が何をする？】

① 店舗オーナー: Rincleで店舗アカウントを登録する
      ↓
② Rincle（自動）: Pay.JPに「この店舗の決済口座を作ってください」と依頼する
   このとき以下の情報を送る:
     - 店舗名
     - 銀行口座情報（振込先）
     - 手数料率: 10%（Rincleの取り分）
     - 最低振込金額: 1,000円
      ↓
③ Rincle（自動）: Pay.JPに「この店舗の審査ページを作ってください」と依頼する
      ↓
④ Rincle（自動）: 審査用URLをブラウザで直接開く
   ※ メール送信ではなく、店舗登録完了後にそのまま画面遷移する
      ↓
⑤ 店舗オーナー: Pay.JPの審査ページで、
   本人確認書類（免許証など）をアップロードする
   ※ この画面はPay.JPが用意したもので、Rincleの画面ではない
      ↓
⑥ Pay.JP（自動）: 提出された書類を審査する（数日かかる場合あり）
      ↓
⑦ Pay.JP（自動）: 審査が終わったら、Rincleに「結果出たよ」と自動通知する
      ↓
⑧ Rincle（自動）: 通知を受け取って、店舗の審査状態を更新する
   - 合格 → 決済受付OK + 店舗オーナーに「審査通過のご案内」メール送信
   - 不合格 → 決済受付NG + 店舗オーナーに「審査否決のご案内」メール送信
```

**ポイント:**
- **審査はPay.JPが行う。** Rincleの管理画面に「承認ボタン」はない
- Rincleは審査結果を自動で受け取るだけ
- 審査に通るまで、その店舗ではクレジットカード決済ができない

**審査状態の一覧:**

| 状態 | 意味 | いつこの状態になる？ | カード決済 |
|---|---|---|---|
| 審査前 | まだ書類を出していない | 店舗登録直後 | 不可 |
| 審査中 | Pay.JPが確認している最中 | 書類提出後 | 不可 |
| 審査通過 | 問題なしと判断された | Pay.JPが承認した時 | **可能** |
| 審査否認 | 不備や問題があると判断された | Pay.JPが否認した時 | 不可 |

### 2. お客さんのカード登録フロー

お客さん（予約する人）がクレジットカードを登録する流れです。予約フローの「支払い方法」ステップ（index/howtopay）で行います。

```
【誰が何をする？】

① お客さん: 予約フローの「支払い方法」ステップに進む
      ↓
② Rincle（自動）: そのお客さんが既にカード登録済みか確認する
   - 登録済み → 登録済みカードを表示（ドロップダウンで選択）
   - 未登録 → カード入力フォームを表示
      ↓
③ お客さん: 「オンライン決済」を選択し、カード情報を入力する
      ↓
④ Pay.JP（自動）: カード入力フォームを表示する
   ※ カード番号はRincleのサーバーを通らない（セキュリティのため）
      ↓
⑤ お客さん: カード番号・有効期限・セキュリティコードを入力する
      ↓
⑥ Pay.JP（自動）: カード情報を受け取り、「トークン」（使い捨ての引換券）を発行する
      ↓
⑦ Rincle（自動）: トークンを使ってカードを登録する
   - 初めてのお客さん → まず顧客アカウントを作成してからカード登録
   - 既存のお客さん → カードを追加登録
      ↓
⑧ Rincle（自動）: 顧客情報を取得して登録完了を確認する
   ※ この時点ではメインカードの設定は行わない（予約確定時に設定される）
```

**ポイント:**
- カード番号はRincleには保存されない（Pay.JPが安全に管理する）
- お客さんは複数のカードを登録できる
- メインカードは**予約確定時**に設定される（カード登録時ではない）

### 3. 予約確定フロー

お客さんが予約を確定するフローです。**この時点ではカード引き落としは行われません。**

```
【誰が何をする？】

① お客さん: Rincleで予約を確定する（支払方法: クレジットカード）
      ↓
② Rincle（自動）: Pay.JPから顧客情報を取得する
      ↓
③ Rincle（自動）: 取得したカード情報の1つ目をデフォルトカードとして設定する
   ※ ドロップダウンで選択したカードではなく、取得した顧客カードの1つ目が使われる
      ↓
④ Rincle（自動）: 予約情報を作成する（ステータス: 来客待ち）
      ↓
⑤ Rincle（自動）: 営業日カレンダーを更新する（自転車を確保）
      ↓
⑥ Rincle（自動）: メールを送信する
   - お客さんに予約確認メール
   - 店舗に予約通知メール
   - PEDALSTANDARD に通知メール
```

**ポイント:**
- 予約確定時に**課金は発生しない**（自転車の確保とカード設定のみ）
- 実際のカード引き落としは、次の「ライド開始時の決済フロー」で行われる

### 3.5. ライド開始時の決済フロー（店舗操作）

店舗スタッフが予約一覧から「ライド開始」を行ったときに、クレジットカード決済が実行されます。

```
【誰が何をする？】

① 店舗スタッフ: shop_admin画面の予約一覧から「ライド開始」をクリック
      ↓
② 店舗スタッフ: 確認ポップアップで「はい」をクリック
      ↓
③ Rincle（自動）: Pay.JPに「このお客さんのカードから○○円を引き落としてください」と依頼する
   このとき以下の情報を送る:
     - 誰の顧客ID？ → お客さんの顧客ID（cus_xxxxx）
     - いくら？ → 合計金額
     - どの店舗に払う？ → 予約先の店舗（ten_xxxxx）
     - Rincleの手数料は？ → 合計金額 × 10%
      ↓
④ Pay.JP（自動）: カードから引き落としを実行する
      ↓
⑤ Rincle（自動）: 決済情報（charge_id）を予約データに保存する
   - エラー時 → 店頭決済（未請求）として更新する
```

**お金の流れ:**

> **重要:** Bubble実装上 `payjp_fee_included: false` が設定されている。
> これは「Pay.JPのカード手数料をプラットフォーム手数料（platform_fee）に含めない」という意味。
> カード手数料は **店舗（テナント）側の取り分から差し引かれる**。

```
例: 予約金額 10,000円 の場合

  お客さんのカード → 10,000円 引き落とし
       ↓
  Pay.JP が一時預かり
       ↓  振込時に分配:
  ├── Rincleへ:  1,000円（platform_fee 10% ← まるまる受け取り）
  ├── Pay.JPへ:    350円（カード手数料 約3.5%）
  └── 店舗へ:   8,650円（10,000 − 1,000 − 350）
```

| 誰が | いくら受け取る | 負担する手数料 |
|------|--------------|--------------|
| **お客さん** | — | 10,000円 支払い |
| **Rincle（PEDALSTANDARD）** | **1,000円**（10%） | なし（`payjp_fee_included: false` のため手数料負担なし） |
| **Pay.JP** | **350円**（約3.5%） | — |
| **店舗** | **8,650円** | platform_fee 10% + カード手数料 約3.5% = **実質 約13.5% 負担** |

> **注意:** 契約上のRINCLEの取り分は **15%** だが、Bubbleの `platform_fee` Option Set は **10** に設定されている。
> 5%分の差額が発生しており、現状RINCLEは本来の取り分より少ない金額しか受け取れていない。

### 3.6. 管理者による決済フロー（admin操作）

管理者（シスアド）が予約一覧のステータスを変更したときにも、決済が実行される場合があります。

```
【誰が何をする？】

① Rincle管理者: admin画面の予約一覧でステータスをドロップダウンから変更する
      ↓
② Rincle（自動）: 条件を満たす場合（来客待ち・未請求 かつ クレジットカード決済）、
   Pay.JPに引き落としを依頼する
      ↓
③ Rincle（自動）: 決済情報を予約データに保存する
```

**ポイント:**
- 現金支払い等で支払いが行われていないユーザーに対して実行する想定
- ドロップダウンのステータス変更で自動発火する（ボタンクリックではない）

### 4. 延長料金の決済フロー

ライド（利用）が予定より長くなった場合、追加料金が発生する流れです。

```
【誰が何をする？】

① 店舗スタッフ: 延長を登録する（合計金額・手数料・延長返却日を更新）
      ↓
② 店舗スタッフ: 「ライド終了」ボタンをクリックする
      ↓
③ Rincle（自動）: 延長料金がある場合、Pay.JPに「追加で○○円を引き落としてください」と依頼する
      ↓
④ Pay.JP（自動）: カードから追加引き落としを実行する
      ↓
⑤ Rincle（自動）: 予約ステータスを決済済みに更新する
   - エラー時 → 返却済み・延長未決済として更新する
```

**ポイント:**
- 延長料金は、ライド開始時の決済とは**別の決済**として処理される
- 返金するときも、それぞれ個別に返金する必要がある

### 5. 返金フロー

予約がキャンセルされたなどの理由で、お客さんに返金する流れです。

```
【誰が何をする？】

① Rincle管理者: admin画面の予約一覧からステータスを「キャンセル」に変更する
      ↓
② Rincle（自動）: キャンセルポップアップで「返金あり」をクリック
      ↓
③ Rincle（自動）: Pay.JPに通常決済の返金を依頼する（charge_idを使用）
      ↓
④ Pay.JP（自動）: お客さんのカードに返金処理を行う
      ↓
⑤ Rincle（自動）: 延長料金の決済があれば（__charge_idが存在する場合）、そちらも返金する
      ↓
⑥ Rincle（自動）: 返金IDを予約データに保存する + 営業日カレンダーを更新する
```

**ポイント:**
- 返金は管理画面からの手動操作（お客さん側からは返金できない）
- 延長料金がある場合は、通常分と延長分の**2回返金**が必要
- 返金されると、店舗への振込額からもその分が差し引かれる

### フロー全体のまとめ

```
【店舗側の準備】
  店舗登録 → Pay.JP審査（数日）→ 審査通過 → 決済受付OK！

【お客さん側の準備】
  カード登録（1回だけ）→ 以降はそのカードで決済OK！

【予約〜決済】
  予約確定（課金なし・自転車確保のみ）
    → 店舗が「ライド開始」→ カード引き落とし → 完了
    ※ 管理者がステータス変更で課金する場合もあり

【延長が発生したら】
  店舗が延長登録 → 「ライド終了」→ 延長料金を追加で引き落とし

【キャンセル・返金】
  管理者が返金実行 → カードに返金（通常分 + 延長分）
```

## お金の流れまとめ（As-Is）

### Pay.JP の設定値

| 設定項目 | 値 | 意味 |
|---------|---|------|
| `platform_fee` | 売上の **10%** | Pay.JPが都度差し引いてRincleに渡す |
| `payjp_fee_included` | **false** | カード手数料はplatform_feeに含めない → **店舗負担** |

### 1件あたりの分配（例: 予約金額 10,000円）

| 受取人 | 金額 | 計算 |
|--------|------|------|
| Rincle（PEDALSTANDARD） | **1,000円** | 10,000 × 10% |
| Pay.JP（カード手数料） | **約350円** | 10,000 × 約3.5% |
| 店舗（ネット入金） | **約8,650円** | 10,000 − 1,000 − 350 |

### 店舗から見た負担率

| 項目 | 率 | 誰に払っている |
|------|-----|--------------|
| プラットフォーム手数料 | 10% | → Rincle（PEDALSTANDARD） |
| カード手数料 | 約3.5% | → Pay.JP |
| **合計負担** | **約13.5%** | |

> **問題:** 契約上のRINCLE取り分は15%だが、実装は10%。5%分の取りっぱぐれが発生中。
