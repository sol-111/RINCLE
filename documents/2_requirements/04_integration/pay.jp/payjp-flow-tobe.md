# クレジット支払いフロー（To-Be）

> クライアント（PEDALSTANDARD 増永様）提供の「RINCLEクレジット支払いフロー.pdf」および定例議事録（3/31, 4/7, 4/10）に基づく。
> 現行実装（As-Is）との差分は末尾に記載。

---

## サマリ

- 予約時に**オーソリ（与信枠確保）のみ**を行い、実際の課金はしない
- レンタル当日、店舗が「ライド開始」で**キャプチャ（決済確定）**する
- キャンセル時は**オーソリ解放**で済み、返金処理が不要になる
- 未回収リスクを抑えつつ、キャンセルしやすい体験を両立する

---

## 登場人物

| 登場人物 | 役割 |
|---|---|
| **エンドユーザー** | バイクを予約・支払いする人 |
| **加盟店（販売店）** | 店舗管理画面でライド開始・精算を行う |
| **PAY.jp** | 決済処理（オーソリ・キャプチャ・返金）を担う |
| **RINCLE** | プラットフォーム。予約管理・在庫管理・ステータス制御 |

---

## 1. 予約〜オーソリフロー

```
【予約時】

① エンドユーザー: RINCLEでバイク予約
      ↓
② エンドユーザー: 支払方法「クレジット」を選択
      ↓
③ RINCLE → PAY.jp: オーソリ（与信枠確保）を実行
   ※ 実際の請求は行わず、利用可能枠のみを一時的に確保
   ※ 未回収リスクを抑えるため、予約時に与信確認のみ実施
      ↓
  【決済エラーの場合】→ エラー表示、別カードで再試行
      ↓
④ PAY.jp: 与信枠確保（有効期限60日、expiry_days=60）
      ↓
⑤ エンドユーザー: 予約確定
      ↓
⑥ RINCLE: 予約確定・在庫確保
```

**ポイント:**
- `POST /v1/charges` を `capture=false` で呼ぶ（オーソリのみ）
- 与信枠の有効期限は**60日間**（`expiry_days=60`）。最大予約先も60日以内に制限する
- この時点でお客さんのカードからお金は引き落とされない

---

## 2. キャンセルフロー

```
【キャンセル時】

① エンドユーザー: キャンセル操作
   ※ 予約前日23時59分までオンラインでキャンセル可能
   ※ それ以降は電話で店舗がキャンセル対応
      ↓
② RINCLE → PAY.jp: オーソリ解放
   ※ まだキャプチャしていないので「返金」ではなく「解放」
      ↓
③ RINCLE: 予約キャンセル・在庫開放
```

**ポイント:**
- オーソリ解放なので、**返金処理が不要**（お金を引き落としていないため）
- キャンセルしやすい体験をサービスとして重視（議事録 4/7: 雨の日でもキャンセルしやすく）
- キャンセル料の請求は基本的に行わない方針（特に現地払いでは回収困難）
- プラットフォーム利用料もキャンセル時は発生しない

---

## 3. レンタル当日 〜 キャプチャフロー

```
【レンタル当日】

① エンドユーザー: 店舗来店
      ↓
② 加盟店: RINCLE店舗管理画面を開く
      ↓
③ 加盟店: 予約情報から「ライド開始」ボタンをクリック
      ↓
④ RINCLE → PAY.jp: キャプチャ（決済確定）
      ↓
  【キャプチャ失敗の場合】
   → 別カードで再決済 OR 別決済手段（現金等）へ切替
   ※ この時点の決済不成立の場合の動きは要定義
      ↓
⑤ 加盟店: 精算（店舗POS）
      ↓
⑥ 加盟店: バイク貸出
      ↓
⑦ エンドユーザー: RIDE!
      ↓
⑧ RINCLE: ステータス変更
```

**ポイント:**
- `POST /v1/charges/:id/capture` でオーソリを確定する
- `platform_fee = 0` のため、RINCLEの利用料はPay.JPでは差し引かれない（月末に別途請求）
- 管理者もadmin画面からステータス変更でキャプチャを実行できる（As-Isの「管理者による決済」に相当）
- キャプチャ失敗時の救済フロー（別カード / 現金）は要定義
- 二重キャプチャ防止の仕組みが必要

---

## 3.5. キャプチャ後の返金フロー

ライド開始後（キャプチャ済み）にトラブル等で返金が必要になった場合のフローです。

```
【キャプチャ後の返金】

① 加盟店 or エンドユーザー: 返金が必要な状況が発生（車体トラブル等）
      ↓
② RINCLE管理者: admin画面から返金操作を行う
   ※ 返金はお客さん側からはできない（管理者のみ操作可能）
      ↓
③ RINCLE → PAY.jp: 返金（refund）を依頼
   ※ キャプチャ済みのため、オーソリ解放ではなく返金APIを使用
      ↓
④ PAY.jp（自動）: お客さんのカードに返金処理を行う
      ↓
⑤ RINCLE: 予約ステータスを更新・在庫を開放
   ※ 返金額はPay.JPから店舗への次回振込から差し引かれる
```

**ポイント:**
- ライド開始前のキャンセル（オーソリ解放）とは異なり、実際にお金を返す必要がある
- As-Isの返金フローと仕組みは同じ（Pay.jpの refund API を使用）
- 月末請求で既にプラットフォーム利用料（15%）を計上済みの場合、**返金分の利用料を翌月請求で調整**する必要がある

---

## 4. お金の流れ（To-Be）

> **As-Isとの最大の違い:** Pay.JPの `platform_fee` を **0** にし、RINCLEの取り分はPay.JP経由ではなく **月末に紙（請求書）で別途請求** する。
> カード手数料は引き続き **店舗負担**（Pay.JPがネット入金時に差し引く）。

### Pay.JP の設定値（To-Be）

| 設定項目 | As-Is（現状） | To-Be（理想） | 変更理由 |
|---------|-------------|-------------|---------|
| `platform_fee` | 売上の **10%** | **0**（差し引かない） | RINCLEの取り分はPay.JP外で請求 |
| `payjp_fee_included` | false | **false**（変更なし） | カード手数料は店舗負担のまま |

### 1件あたりの分配（例: 予約金額 10,000円）

```
  エンドユーザーのカード → 10,000円 キャプチャ（確定）
       ↓
  PAY.jp が一時預かり
       ↓
  ├── Pay.JPへ:    350円（カード手数料 約3.5%）
  └── 加盟店へ:  9,650円（ネット入金 = 10,000 − 350）
       ↓
  【Pay.JPの外で別途処理】
  加盟店 → PEDALSTANDARD: 1,500円（売上10,000円 × 15% を月末に請求書で請求）
```

| 受取人 | 金額 | 計算 | いつ届く |
|--------|------|------|---------|
| Pay.JP（カード手数料） | **約350円** | 10,000 × 約3.5% | Pay.JP振込時に自動差引 |
| 加盟店（ネット入金） | **約9,650円** | 10,000 − 350 | Pay.JP振込日（月2回 or 月1回） |
| PEDALSTANDARD（利用料） | **1,500円** | 10,000 × 15% | **月末に別途請求書で回収** |

### 店舗から見た負担率

| 項目 | As-Is（現状） | To-Be（理想） | 誰に払う |
|------|-------------|-------------|---------|
| カード手数料 | 約3.5% | **約3.5%**（変更なし） | → Pay.JP |
| プラットフォーム利用料 | 10%（Pay.JP経由） | **15%**（月末請求書） | → PEDALSTANDARD |
| **合計負担** | **約13.5%** | **約18.5%** | |

> **注意:** 店舗から見ると、As-Is（約13.5%）→ To-Be（約18.5%）で **負担が5%増える**。
> ただし As-Is の10%は契約上15%であるべきなので、本来の契約通りに是正すると考えれば実質的な増分はカード手数料の3.5%のみ。
> As-Isでは platform_fee 経由で自動徴収されていたが、To-Be では月末請求に変わるため、**未払いリスク** が新たに発生する点に留意。

### As-Is → To-Be の変化（お金の流れ）

| 項目 | As-Is | To-Be |
|------|-------|-------|
| RINCLEの受取方法 | Pay.JPが自動で分配 | **月末に請求書で別途回収** |
| RINCLEの受取率 | 10%（実装ミス。契約上は15%） | **15%** |
| カード手数料の負担者 | **店舗** | **店舗**（変更なし） |
| 店舗へのネット入金 | 約8,650円/万円 | **約9,650円/万円**（Pay.JPからの入金は増える） |
| 店舗の実質負担 | 約13.5% | **約18.5%**（契約上は同等） |
| 未払いリスク | なし（自動徴収） | **あり**（月末請求の回収が必要） |

| 項目 | To-Be |
|------|-------|
| カード売上の帰属 | **加盟店**がPAY.jpからネット入金を受ける |
| プラットフォーム利用料 | 売上の**15%**をPEDALSTANDARDへ**毎月末請求**（Pay.jpの`platform_fee`では抜かない） |
| Pay.jp手数料 | カード利用料 約3.5%（Pay.jpが差し引き。**店舗負担**） |
| オーソリ有効期限 | **60日**（`expiry_days=60`）。最大予約先を60日以内に制限 |
| 営業カレンダー | 365日分 → **60日分**に縮小（オーソリ期限と整合） |

## 4.5. 月末プラットフォーム利用料の請求フロー

RINCLEの利用料（売上の15%）は、Pay.JP経由ではなく**毎月末に請求書で別途回収**する新しいフローです。

```
【月末請求】

① RINCLE（自動 or 手動）: 月末に各加盟店のキャプチャ済み売上を集計
   ※ 返金が発生した分は売上から差し引いて集計
      ↓
② PEDALSTANDARD: 売上合計 × 15% の請求書を作成し、加盟店に送付
      ↓
③ 加盟店: 請求書に基づき指定口座へ振り込む
      ↓
④ PEDALSTANDARD: 入金確認
   ※ 未入金の場合 → 督促対応（運用ルールは要定義）
```

**ポイント:**
- As-IsではPay.JPが自動で手数料を差し引いていたため、未払いリスクはゼロだった
- To-Beでは請求書ベースの回収になるため、**未払い・遅延リスク**が新たに発生する
- 締め日・入金期限・督促ルールの策定が必要

## 5. 現地払いフロー

```
【現地払いの場合】

① エンドユーザー: 支払方法「現地払い」を選択
      ↓
② RINCLE: 予約確定・在庫確保（PAY.jp APIは呼ばない）
      ↓
③ エンドユーザー: 来店時に店舗で精算
      ↓
④ RINCLE: ステータスを精算完了に変更
```

**ポイント:**
- Pay.jpのChargeは作成しない（`charge_id`は空）
- テナント設定ミスによる誤入金は発生しない
- キャンセル料の回収は現実的に困難（議事録 4/7: 常連さん以外はほぼ回収できない）

## 7. As-Is との差分

### 決済フローの差分

| 項目 | As-Is（現状） | To-Be（理想） |
|------|--------------|--------------|
| **予約時** | カード設定のみ（課金なし、オーソリもなし） | **オーソリ（与信枠確保）** |
| **ライド開始時** | `POST /charges` で全額直接課金 | **`POST /charges/:id/capture`** でオーソリを確定 |
| **キャンセル時** | admin が `POST /charges/:id/refund` で返金 | **オーソリ解放**（返金不要） |
| **決済失敗時** | 店頭決済（未請求）として更新 | 別カードで再決済 or 別手段へ切替 |
| **キャプチャ失敗時** | 該当なし（直接課金のため） | 別カード / 現金フォールバック（**要定義**） |
| **キャプチャ後の返金** | admin が返金処理を実行 | admin が返金処理を実行（同様）+ **利用料の調整が必要** |
| **管理者によるキャプチャ** | ステータス変更で全額課金 | ステータス変更で**オーソリをキャプチャ** |
| **利用料の回収** | Pay.JPが自動で差し引き（10%） | **月末に請求書で回収（15%）** |

### お金の流れの差分

| 項目 | As-Is（現状） | To-Be（理想） |
|------|--------------|--------------|
| **platform_fee（Pay.JP上）** | **10%**（Pay.jpが都度自動差し引き） | **0**（Pay.jpでは差し引かない） |
| **RINCLE利用料の徴収方法** | Pay.JPが自動分配 | **月末に請求書で別途請求（15%）** |
| **RINCLE利用料の率** | 10%（※契約上は15%。実装ミス） | **15%** |
| **カード手数料（約3.5%）の負担者** | **店舗**（`payjp_fee_included: false`） | **店舗**（変更なし） |
| **店舗のネット入金（1万円あたり）** | 約8,650円 | **約9,650円**（Pay.JPからの入金は増える） |
| **店舗の実質負担率** | 約13.5%（10% + 3.5%） | **約18.5%**（15% + 3.5%）※契約上は同等 |
| **RINCLE側の未払いリスク** | なし（自動徴収） | **あり**（月末請求の回収が必要） |
