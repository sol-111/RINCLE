# Stripe移行方針（2026/4/21 MTG決定）

> 4/21 定例MTGでの増永氏との合意に基づく。
> 既存の Pay.jp To-Be（オーソリ方式）から方針転換。

---

## 背景・なぜ移行するか

### Pay.jp の課題
- プラットフォームタイプの制約で、**店舗ごとに Pay.jp アカウント作成が必要**
- 販売店側から見ると「請求される」「アカウント登録が面倒」で評判が悪い
- 一括回収モデルが実現可能か不明（Pay.jp に問い合わせ中だが回答未定）
- 現行実装にセキュリティ上の問題あり（APIキー・顧客情報がDB上で丸見え）

### Stripe 移行のメリット
- **店舗に決済アカウントを作らせる必要がない**（Connect 不使用）
- リンクルが一括で売上を回収し、手数料を差し引いて店舗に振り込むシンプルなモデル
- 予約時決済にすればオーソリ60日問題が消える
- Bubble との連携実績が豊富

---

## 新しい決済モデル概要

```
【お金の流れ】

エンドユーザー → Stripe → PEDALSTANDARD（一括回収）
                                    ↓
                          カード手数料 3.25%（Stripe手数料）差引
                                    ↓
                          プラットフォーム利用料 15% 差引
                                    ↓
                              店舗へ振込
```

### 1件あたりの分配（例: 予約金額 10,000円）

| 受取人 | 金額 | 計算 |
|--------|------|------|
| Stripe（カード手数料） | 約325円 | 10,000 × 3.25% |
| PEDALSTANDARD（利用料） | 1,500円 | 10,000 × 15% |
| 店舗（振込額） | 8,175円 | 10,000 - 325 - 1,500 |

> Stripe の手数料は 3.25%（国内カード）。Pay.jp の 3.5% より若干安い。
> 手数料率は要確認（プランにより変動あり）。

### Pay.jp（現行）との比較

| 項目 | Pay.jp（現行） | Stripe（新） |
|------|---------------|-------------|
| 店舗のアカウント作成 | **必要** | **不要** |
| 売上の帰属先 | 店舗（Pay.jpが分配） | **PEDALSTANDARD（一括回収）** |
| プラットフォーム利用料 | 10%（Pay.jp自動差引）+ 5%紙請求 | **15%（PEDALSTANDARD側で差引後振込）** |
| カード手数料 | 約3.5%（店舗負担） | 約3.25%（PEDALSTANDARD負担 → 店舗振込額で調整） |
| 未払いリスク | あり（紙請求5%分） | **なし（一括回収のため）** |
| オーソリ最大日数 | 60日 | 7日（Stripe標準） / 31日（延長） |

## 決済フロー

### 1. 予約時決済（通常バイク）

```
① エンドユーザー: バイク予約 + カード情報入力
      ↓
② RINCLE → Stripe: PaymentIntent 作成 + 即時決済（confirm=true）
      ↓
  【決済エラー】→ エラー表示、別カード or 再試行
      ↓
③ Stripe: 決済完了
      ↓
④ RINCLE: 予約確定・在庫確保
```

- 予約時に全額決済する（オーソリではない）
- 現金決済は廃止。カード決済オンリー

### 2. 高級車のデポジット（オーソリ）

```
① エンドユーザー: 高級車を予約 + 通常料金を即時決済
      ↓
② RINCLE → Stripe: デポジット分のオーソリ（例: 10万円）
   ※ capture_method=manual で与信枠のみ確保
      ↓
③ 貸出時: 問題なければオーソリ解放
   ※ 車体損傷等あればキャプチャして回収
```

- デポジットオーソリは最大7日で十分（増永氏確認済み）
- 通常バイクにはデポジット不要

### 3. キャンセルフロー

```
① エンドユーザー or 店舗: キャンセル操作
      ↓
② RINCLE → Stripe: 返金（Refund）
   ※ キャンセルポリシーに基づき全額 or 一部返金
      ↓
③ RINCLE: 予約キャンセル・在庫開放
```

- 予約時決済のため、キャンセル時は**返金処理が必要**（オーソリ解放ではない）
- キャンセルポリシーの策定が必須（下記「決め残し」参照）

### 4. 延長決済

```
① 店舗: 延長操作（延長時間・追加料金を確定）
      ↓
② RINCLE → Stripe: 追加の PaymentIntent を作成・決済
   ※ 登録済みカード（PaymentMethod）を再利用
      ↓
③ RINCLE: 延長情報を更新
```

- 元の予約とは別の PaymentIntent として処理
- 予約IDとの紐付けをDB上で管理

### 5. 現場での追加オプション

```
① 店舗: 追加オプション選択（ヘルメット、保険等）
      ↓
② RINCLE → Stripe: 追加決済（登録済みカード使用）
      ↓
③ RINCLE: オプション情報を更新
```

- 現在はリンクル外で処理されている（現場で別途決済）
- システム上で管理できるようにすることで、売上の把握精度が向上

## 店舗への振込フロー

```
【月次振込】

① RINCLE（自動）: 月末に各店舗のStripe決済済み売上を集計
      ↓
② 集計: 売上合計 - Stripe手数料 - プラットフォーム利用料(15%) = 店舗振込額
      ↓
③ PEDALSTANDARD: 店舗へ振込
```

- Pay.jp 時代の「紙の請求書で5%を追加徴収」が不要になる
- 未払いリスクがゼロ（先に全額回収しているため）
- 振込頻度・締め日は要決定（月1回 or 月2回）

## 移行ステップ

### Phase 0: 即時対応（GW前）
- [x] プラットフォーム手数料を 10% → 15% に変更（Pay.jp上）
- [ ] テスト環境の本番キー使用を修正
- [ ] DBのセキュリティ対応（APIキー・顧客情報の非表示化）
- [ ] 延長決済の処理不備修正
- [ ] カード選択の不具合修正

### Phase 1: Stripe導入準備（GW明け〜）
- [ ] 弁護士回答の確認（売上計上・責任所在の整理）
- [ ] Stripeアカウント開設・本番キー取得
- [ ] Stripe決済の詳細仕様設計
- [ ] キャンセルポリシーの確定（増永氏側）
- [ ] DB設計の見直し（決済関連テーブル）

### Phase 2: Stripe実装
- [ ] Bubble × Stripe 連携の実装（PaymentIntent / PaymentMethod）
- [ ] 予約時決済フローの実装
- [ ] キャンセル・返金フローの実装
- [ ] 延長決済フローの実装
- [ ] 高級車デポジットオーソリの実装
- [ ] 管理画面への決済ステータス表示

### Phase 3: 移行・切替
- [ ] テスト環境でのE2E検証
- [ ] 既存ユーザーのカード情報再登録の案内（Pay.jp → Stripe）
- [ ] Pay.jp → Stripe 切替（本番）
- [ ] Pay.jp の店舗アカウント廃止
- [ ] 店舗への振込フロー運用開始

## 決め残し・要確認事項

### 増永氏側で決定が必要
- [ ] キャンセルポリシーの策定（何日前まで全額返金、当日は○%等）
- [ ] 弁護士回答: 売上を一括回収する場合の会計上の扱い・責任所在
- [ ] 店舗への振込頻度（月1回 / 月2回）
- [ ] 振込手数料の負担者（PEDALSTANDARD / 店舗）
- [ ] 追加オプション（ヘルメット等）をシステム化するか、引き続き現場対応か

### 技術的な確認事項
- [ ] Stripe手数料率の確認（プラン・契約条件による）
- [ ] Bubble × Stripe プラグインの選定・検証
- [ ] 既存の Pay.jp 顧客データ（カード情報）の移行方法
  - カード情報自体は PCI DSS の制約で直接移行不可 → ユーザーに再登録してもらう必要あり
- [ ] Webhook の設計（決済成功/失敗/返金の通知受信）
- [ ] 3Dセキュア対応（Stripe は標準対応だが、Bubble側のフロー設計が必要）
